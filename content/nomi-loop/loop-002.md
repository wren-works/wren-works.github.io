### 案例D："全舱免唤醒"的渐进式路线——从一刀切到精细化

- **来源**：NIO官方版本说明书（2024.11）+ Banyan 3.1.0更新日志（2025.01）
- **2024.11**：Banyan 3.0.0 推出"全舱免唤醒"，但**需要用户主动去设置里打开开关**（非默认）
- **2025.01**：Banyan 3.1.0 新增"主驾免唤醒模式"——仅主驾支持免唤醒，副驾/后排仍需"Hi NOMI"
- **同时**：全舱免唤醒与连续对话开关解绑，提升免唤醒响应成功率
- **解读**：蔚来的免唤醒策略不是"一刀切全开"，而是一条谨慎的渐进路线。先让用户主动开启（作为feature而非默认），再进一步缩窄为"仅主驾"——这暗示**全舱免唤醒可能存在副驾/后排误触发问题**，团队对此在做精细化控制。行业数据显示2025年"免唤醒"渗透率已达60.6%，已是标配。蔚来有功能但做得很克制。

### 2. 推理 — 交互失败在哪层？

**核心推理：Loop #001 的洞察需要大幅修正**
- Loop #001 认为"免唤醒是NOMI最大短板，需优先解决"
- 实际：免唤醒已在2024年上线，但采用的是**需要手动开启 + 后续缩窄为仅主驾**的谨慎路线
- 这说明问题层级发生了变化：不是"有没有免唤醒"，而是"免唤醒体验够不够好"→ 涉及ASR误触发率、多模拒识在实际场景中的表现

**新问题的技术定位：**
- **ASR层**：全舱免唤醒下，副驾/后排的对话是否会被误识别为指令？Banyan 3.1.0推出"主驾免唤醒模式"反向证明了这个问题的存在
- **NLU/拒识层**：多模拒识的96.8%准确率是在什么场景下测的？如果是在副驾闲聊场景中，误响应率可能仍然让用户在意
- **对话管理层**：全舱免唤醒开关与连续对话最初绑定，3.1.0才解绑——说明产品设计中存在耦合过度的问题
- **产品架构层**：功能有但需要"主动开启"才生效，这与竞品（理想、小鹏）的默认免唤醒体验存在认知落差

**2023→2025时间线更新：**
```
2023.02 → 素材2：用户痛骂"叫不醒、乱识别、差小鹏10年"
2023-2024 → 多模拒识模型开发、训练（12000h语音+2000万文本）
2024.09 → 多模拒识技术解读发布，准确率96.8%
2024.08 → Banyan 3.0 发布，全舱免唤醒上线（需手动开启）
2024.11 → Banyan 3.0.0 正式推送，含停车场自由搜（修复了Loop#001发现的不支持停车场搜索问题！）
2025.01 → Banyan 3.1.0：主驾免唤醒模式、全舱免唤醒开关解绑、超拟人情感音色
2025.03 → 盖世论坛：雷鸣介绍NOMI具备"语音免唤醒、多人自由说"
2025-2026 → 公开搜索几乎找不到新的"叫不醒/乱识别"大规模吐槽
```

### 4. 执行 — 一句话洞察

> **如果我是NOMI的PM，我会优先让免唤醒"默认开启且无感好用"——不是因为竞品都有所以我也要有，而是因为NOMI已经走完了"做出来→手动开启→缩窄为仅主驾"三阶段渐进路线，这说明技术底座存在但体验信心不足。** 全舱免唤醒需要手动去设置里翻开关，大部分用户根本不知道有这个功能；主驾免唤醒模式的出现反向证明了误触发问题还没解决到让团队放心全量推送。在行业免唤醒渗透率已超60%的2025年，NOMI的免唤醒体验不是"有没有"的问题，而是"敢不敢默认开"的问题——这已经从功能缺失演变成了体验信心缺失。接下来的战场不是ASR准确率再提升2%，而是让96.8%的多模拒识准确率转化为"用户不用操心设置、坐上车就能自然说话的默认体验"。
