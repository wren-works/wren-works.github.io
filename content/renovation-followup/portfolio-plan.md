能力说明建议：

## Case 页主线

### Hero

标题：FollowUp Copilot

主张：

> 不是替负责人自动追客户，  
> 而是让每次跟进重新建立在上下文上。

状态只写“负责人受控试用中”，不要写“已上线”或“已验证有效”。

### 01 · 真实问题

从一条具体观察开始：装修客户决策周期长，信息散在聊天、附件和线下沟通里；重新跟进时需要重翻记录。补充“客户不回复”不是一句销售话术可以解决的问题。

### 02 · 问题重构

展示项目从“自动生成设计 Brief”转向“客户跟进 Copilot”的过程。强调为何没有直接做完整 CRM、个人微信接入或自动发送。

### 03 · 产品链路

用一条流程说明：

```text
聊天截图 / 脱敏文本
→ 可编辑转写
→ 客户状态卡
→ 证据与冲突
→ 下一步建议
→ 人工确认并发送
```

### 04 · 三个关键决策

1. **证据优先**：重要结论必须回到消息 ID；
2. **规则与模型分层**：确定字段由规则处理，语义判断交给 API；
3. **公开与私有分离**：作品集 prototype 使用脱敏样本，真实资料只进入受控私有版本。

### 05 · 从五组数据到真实试用

可公开的数字：

- 5 组历史客户案例；
- 23 张原始截图；
- 193 条可见消息；
- 1 份负责人确认的 Gold 标准答案；
- 已进入负责人真实试用。

不要公开客户姓名、地址、金额、原始聊天或私有应用链接。不要把“开始测试”写成效果结论。

### 06 · 当前结果和下一轮验证

已完成：历史数据整理、规则层、模型 API、截图转写、人工校对、私有保存和远程受控试用。

下一轮重点：

- 恢复上下文需要多久；
- 负责人修改了哪些高影响字段；
- 跟进建议是否被采用；
- 人工导入是否成为持续使用阻力。

## 媒体建议

公开页面使用 prototype 的脱敏示例，不使用私有版真实数据。

建议准备三张桌面端界面图：

1. 导入脱敏聊天 / 截图后的可编辑转写；
2. 客户阶段、需求、报价和冲突状态卡；
3. 证据定位、下一步建议和人工确认。

可选增加一段 25～35 秒录屏，只演示：导入脱敏示例 → 生成客户卡 → 点击证据 → 修改并确认。录屏中不要出现模型密钥、真实客户列表、浏览器历史或私有隧道地址。

## API 与公开边界

- 不把 `.env.local`、API Key、私有数据库和真实输出复制到作品集；
- 不在 GitHub Pages 上直接提供可调用的模型接口；
- Case 页只说明“服务端调用兼容 API，并使用严格 Schema 校验”；
- 如果以后提供公开 Demo，需要单独增加限流、预算上限、身份验证和合成输入；
- 当前最稳妥的形式是静态 Case 页，不公开真实试用入口。
