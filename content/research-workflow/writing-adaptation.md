# khazix-writer → 产品作品集写作适配

参考来源：`KKKKhazix/khazix-skills/khazix-writer`。目标不是模仿作者人格，而是提取能够降低阅读负担、增强真实感的写作机制。

## 保留的机制

| 原方法 | 在作品集中的用法 |
| --- | --- |
| 从具体事件开始 | 从 NOMI 研究差点被早期假设带偏写起，不从“AI 研究很重要”开始 |
| 亲自下场 | 写清我做了 Skill、跑了同模型对照、盲审并排除污染样本 |
| 层层剥开 | 现象 → 表面解释 → 更深问题 → 产品判断，不先把所有结论列完 |
| 失败作为转折 | 厚 Skill 没增加强洞察，成为拆分两个 Skill 的原因 |
| 逐一展示 | 按会议助手、智能手环、具身智能、欧盟架构逐轮推进，不只放总表 |
| 短段落与句式断裂 | 重要判断单独成段，让数字和转折有呼吸空间 |
| 扣主线句 | 每个实验后回答“这改变了我对 Skill 的什么判断” |
| 承认不确定 | 明确开发集、单次运行、无 token 和无真实决策结果 |
| 开头结尾回环 | 从“想让 NOMI 研究更好”开始，以“得到的是可证伪工作流”收束 |

## 不照搬的部分

- 不模仿粗口、网络口癖和夸张标点；
- 不取消小标题，招聘方需要快速扫描；
- 不强行加入文化或哲学升华；
- 不虚构情绪、对话、人物和现场细节；
- 不使用公众号求赞、星标和关注结尾；
- 不把篇幅、热闹程度和“活人感”置于证据准确性之前；
- 不照搬对冒号、引号或 bullet list 的机械禁令。

## Case 页段落规则

每一节尽量遵循：

```text
一个具体发现
→ 当时的直觉或预期
→ 证据发生了什么
→ 我因此改变了什么判断
```

正文段落以 1–4 句为主。表格只放真正需要横向比较的数据，解释放回叙事里。每个技术名词第一次出现时，用一句普通话解释。

## 推荐页面节奏

1. 具体失误开场；
2. 为什么“更像报告”不是目标；
3. 如何把有用操作化；
4. 第一轮失败；
5. 污染样本与实验纪律；
6. 拆成两个 Skill；
7. 两组最有代表性的结果；
8. 当前能说什么、不能说什么；
9. 回扣开头。
