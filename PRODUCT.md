# AI 产品经理作品集：产品定位与维护规范

这是一个面向 AI 产品经理 / 产品经理岗位的个人作品集网站。招聘方应能通过三个案例快速理解：我如何定义问题、做产品取舍、把 AI 能力放进真实工作流，以及如何区分已经验证的结果和仍待验证的假设。

## 目标

- 用少量完整案例呈现产品思考和执行能力；
- 展示从独立原型、产品研究到真实业务试用的不同工作类型；
- 让重要结论有事实、样本或公开来源支撑；
- 保持结构简单，适合持续更新并直接发布到 GitHub Pages；
- 为 AI 产品经理 / 产品经理岗位提供比简历更完整的决策过程。

## 当前主线

> AI 如何进入真实工作流，同时保留人的判断、控制感和纠错空间。

三个项目分别回答不同问题：

- **FollowUp Copilot**：能否把 AI 产品推进到真实业务试用，并处理隐私、证据和人工审核边界；
- **DecisionHelp**：能否从个人真实问题出发，完成移动端产品定义、实现和核心链路验证；
- **AI Research Workflow**：能否把研究 Skill 的价值从主观感受变成可比较、可失败和可更新的实验问题。

NOMI 研究保留为问题起点：早期假设差点带偏后续搜索，促使我把研究流程封装并做同模型对照。当前形成 `explore-topic` 与 `decision-research` 两个 Skill，分别优化开放探索和固定选项决策；现有结果只属于开发集早期信号，不主张普遍优于默认模型。

```text
  │   └── AI Research Workflow
  └── Footer

  ├── 问题与目标
  ├── 核心工作流 / 研究过程
  ├── 关键产品决策
  ├── 当前结果与证据
  ├── 局限和下一步
  └── 下一项目
```

首页只负责帮助招聘方判断“值不值得点进去”；完整过程放在独立 Case 页，不在首页堆叠功能和技术细节。

## 首页项目顺序

| 顺序 | 项目 | 类型 | 主要展示能力 | 当前状态 |
|---|---|---|---|---|
| 1 | FollowUp Copilot | AI Product · Real-world Pilot | 问题重构、工作流设计、证据机制、AI API、隐私边界、真实试用 | 负责人受控试用中 |
| 2 | DecisionHelp | AI Product · Android App | 产品定义、解析策略、架构设计、实现验证 | 个人真实场景验证 |
| 3 | AI Research Workflow | AI Workflow · Skill Evaluation | 问题路由、Skill 设计、对照实验、证据与可更新性 | 开发集验证，跨任务复现待补 |

排序原则：优先展示证据更强、与目标岗位更相关、离真实使用更近的项目，而不是按创建时间排序。

## 首页卡片规范

每张卡片包含：

- 项目类别 Tag；
- 项目形态或验证状态；
- 项目名称；
- 一句话说明用户问题和产品作用；
- 能力关键词；
- 阅读时间与 Case 链接。

卡片描述不写功能清单，也不使用“赋能、颠覆、显著提升”等无法验证的表达。

## Case Study 写作规范

Case 页不强制使用完全相同的章节名称，但必须回答六类问题：

1. **Context**：项目为什么出现，面对的是谁的什么问题；
2. **Goal**：这一阶段真正要验证什么，明确不做什么；
3. **Approach**：如何拆解问题，为什么选择当前方案；
4. **Decisions**：有哪些关键取舍，放弃了哪些看似自然的方向；
5. **Evidence**：已经观察到什么，数据、样本或来源是什么；
6. **Limits / Next**：哪些结论还不能成立，下一轮优先验证什么。

### 叙事原则

- 先写决策链路，再写功能列表；
- 事实、推断和假设分开；
- “开始试用”不能写成“已经验证有效”；
- 个人使用不能写成公开用户研究；
- 公开资料研究不能暗示内部合作或任职关系；
- AI 辅助实现可以说明，但产品决策、架构和验证责任要写清楚；
- 失败、复杂度和未完成项可以保留，它们应指向下一轮产品判断。

### 视觉与媒体原则

- 媒体只用于证明关键链路或帮助理解产品，不为填满页面而添加；
- 真实界面仍在变化时，可以使用标注清楚的脱敏重构示意；
- 录屏控制在约 25–40 秒，只覆盖一条核心任务链路；
- 不自动播放视频；
- 图片、视频必须有说明文字和可理解的替代文本；
- 不公开真实客户身份、地址、报价、聊天原文、私有入口或密钥。

## 三个 Case 的展示重点

### FollowUp Copilot

主线：从五组历史聊天定义客户状态卡，再推进到负责人真实试用。

必须保留：

- 从“设计 Brief”转向“客户跟进 Copilot”的问题重构；
- 证据 ID、不确定性和冲突处理；
- 规则层与模型 API 的职责分工；
- 作品集 prototype 与私有使用版的隔离；
- 首轮反馈：“识别较准、字段几乎不用改，但操作稍复杂”；
- 当前只是早期信号，不能宣称提高回复率、效率或成交率。

### DecisionHelp

主线：把跨平台收藏截图转成可编辑、可比较、可缩减的地点候选。

必须保留：

- 截图作为低成本输入；
- OCR / Parser 失败如何被用户修正；
- 重复地点由用户确认，不自动合并；
- 同一起点的驾车和公共交通时间；
- 个人真实场景验证与尚无公开用户测试的边界。

### AI Research Workflow

主线：从一次研究偏航出发，把开放探索与固定选项决策拆成两个 Skill，再用同模型对照检验它们到底改变了什么。

必须保留：

- NOMI 只作为问题起点，不能声称新版 Skill 改善了原研究；
- 第一版厚 Skill 没有带来更多强洞察，并显著增加耗时；
- 污染样本被保留但排除，实验过程本身需要可审计；
- `explore-topic` 与 `decision-research` 优化的是两种不同认知动作；
- 当前仅有开发任务、单次运行，没有 token、成本和真实决策结果；
- 页面突出改变产品判断的证据，小幅评分和完整数字留在实验记录。

## 技术与发布边界

- 网站继续使用原生 HTML、CSS 和少量 JavaScript；
- 不引入 npm、打包工具或前端框架；
- 使用系统字体，不依赖外部字体服务；
- 所有页面使用相对路径，兼容 GitHub Pages 项目站点；
- Markdown 可以作为 Brief、研究记录和支撑材料，但当前正式 Case 页以静态 HTML 为准；
- 修改后至少检查首页、三个 Case 页、简历链接、媒体链接和项目间导航；
- API、数据库和真实业务应用不属于作品集网站的运行依赖。

## 隐私与公开规则

作品集只允许出现：

- 脱敏或人工重构的示例；
- 无法反向识别个人的聚合数字；
- 已公开的官方资料；
- 明确标注来源和局限的研究结论。

禁止进入作品集：

- API Key、`.env.local`、私有数据库；
- 真实客户姓名、头像、微信号、电话和详细地址；
- 原始聊天截图、精确报价、合同和户型文件；
- 私有试用地址、登录密码和临时隧道链接；
- 未经确认的负责人或客户原话。

## 新增或更新项目

### 新增项目

1. 先在 `content/<project>/` 中完成 Project Brief；
2. 明确可公开事实、验证状态和隐私边界；
3. 创建独立 Case 页和项目样式；
4. 在首页按证据强度和岗位相关性决定排序；
5. 更新项目间“下一项目”导航；
6. 同步更新 `README.md` 和本文件；
7. 发布前检查相对链接和公开素材。

### 更新现有项目

- 新反馈优先更新“当前结果 / 局限 / 下一步”，不要重写已经发生的历史；
- UI 变化只在影响产品判断时进入正文；
- 新增数字前记录口径与样本范围；
- 真实试用反馈应区分原话、概括和行为记录；
- 只有获得重复使用和对照结果后，才把“试用中”升级为“已验证”。

## 设计约束

- 首页最大阅读宽度约 760px，Case 页约 880px；
- 浅色、留白和系统字体为主；
- 色彩服务于信息层级，不超过少量主色；
- 不使用花哨入场动画；
- 不自动播放媒体；
- 桌面与手机均应保持可读；
- 内容优先于装饰，真实边界优先于完整故事。
